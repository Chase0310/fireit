## 痛点：AI 时代的开发，是一场"黑箱接力"

今天的 AI 辅助开发，产品、开发、AI 三方在一条**黑箱接力链**上互相投喂：

```
产品写 PRD ──喂──▶ 开发 ──喂──▶ AI 写代码 ──出 bug──▶ 开发 ──喂──▶ AI 解释
  (AI 写的，        (不懂)        (不懂，                 (不懂)        (AI 可能也
   自己不理解)                     被动执行)                            不记得了)
```

链路里每个箭头都是一次**信息损耗 + 理解断层**。真正的"理解者"和"执行者"都是 AI，**人退化为传话筒**。

后果是三个失：

- **看不懂** — PRD 是 AI 写的、代码是 AI 写的，人不是"想出来"的，是"收到"的
- **不掌控** — 出问题不知道在哪，无法定位、无法干预
- **双向失忆** — 人靠 AI 解释、AI 也丢了上下文，互相问，互相猜，**一团糟**

> 这就是今天的 AI 开发现状——**一场胡闹厨房（Overcooked）**：每个人都在手忙脚乱传东西，到处救火，没有谁真正懂这道菜怎么做出来的。
>
> 不是热闹好玩，是**乱**。

---

## 北极星：把人重新拉回 loop

> **fireit 不是"更高效地调度 AI"，而是"把人重新拉回开发的主导位置"。**

一句话定位：

> **把人和 AI 拉进同一个共享工作空间——消除"喂给 AI"的传话环节，让人只在必要的时刻介入。**

这个北极星由**两个不可分割的支柱**支撑。缺任何一个，fireit 就退化成又一个"调 LLM 的工具"。

---

## 支柱一：共享工作空间，消除"喂给 AI"

### 问题
现在的 AI 是**被召唤的外部工具**——你把上下文喂给它，它吐结果，结束。下次又要重新喂。上下文在人和 AI 之间反复搬运，每次都是损耗。

### 愿景
**所有 AI 都在同一个共享工作空间里持续在场。** PRD、代码、bug 状态、设计决策——对这些 AI 来说是**直接可见的共享现实**，不需要人来搬运。

```
现在：人 ──喂──▶ AI   （AI 是外部工具，人当传话筒）

fireit：人 + AI1 + AI2 ... 都在同一个工作空间
        └─ PRD、代码、状态对所有人/AI 共享可见
           └─ 没有箭头，因为没有"喂"这个动作
```

### 这意味着什么（设计约束）

| 约束 | 含义 |
|------|------|
| **上下文共享是地基，不是优化项** | 同一订单/工作区内，所有 agent 共享上下文，任何人都不重述 |
| **AI 是持续在场的参与者** | 不是被召唤的工具，是一直在厨房里的厨师 |
| **状态是共享现实** | "这道菜到哪了""bug 在哪"对所有在场者同样可见，不需要人转述 |
| **性格和身份是手段** | 让 AI 成为"可信任的在场者"——你要信任的是队友，不是要反复解释的外部工具 |

---

## 支柱二：人在必要的时间介入

### 问题
两个极端都坏：
- **全程盯着** → 累，注意力被"等待 AI"耗光
- **完全甩手** → 失控，看不懂产出，出问题救不了火

### 愿景
**系统判断"什么时候该叫人"，并在该叫的时候把你拉回来。** 其余时间人可以离开，AI 自主推进；需要你时，系统带着完整的上下文把你拉回决策点。

### 这意味着什么（设计约束）

| 约束 | 含义 |
|------|------|
| **介入是事件驱动的，不是持续监控** | 人不需要盯着，系统在决策门/品菜/不可逆操作时主动叫人 |
| **介入点必须带完整上下文** | 叫你回来时，你看到的是"这道菜为什么卡住、改了什么、要你决定什么"——不是"请你输入" |
| **介入是有意义的，不是噪音** | 只有真正需要人决策的时刻才叫人（呼应 Raft：沉默也是协作） |
| **人保留最终拍板权** | 最终验收、不可逆操作必须人决策——"final acceptance is a user decision" |

## 两个支柱的关系

**它们是一个整体的两面，不可分割：**

- 只有支柱一（共享空间）没有支柱二（必要介入）→ 一个高效的黑箱，人还是不懂、不掌控
- 只有支柱二（必要介入）没有支柱一（共享空间）→ 每次介入都要重新喂上下文，介入本身变成负担
- **两个一起** → 人从"传话筒"回到"主导者"：不用喂（共享空间），不用全程盯（必要介入）

```
        支柱一：消除"喂给 AI"          支柱二：必要时刻介入
        （AI 持续在场，上下文共享）      （人只在决策点被拉回）
                 ╲                 ╱
                  ╲               ╱
                   ╲             ╱
                    ╲           ╱
                     ╲         ╱
                      ╲       ╱
                       ╲     ╱
                        ╲   ╱
                         ╲ ╱
                          ▼
                   人重新成为
                  开发的主导者
```

## 厨房隐喻的真正含义

`fireit` 的厨房 brigade 不是为了热闹好玩。**它是对"有序 vs 胡闹"的回答。**

| | 胡闹厨房（现状） | fireit 的厨房（目标） |
|---|---|---|
| 上下文 | 每人各有一份，靠吼（喂）传递 | **共享的出菜口，对所有人在场者可见** |
| 进度 | "那道菜到哪了？谁在做？" 谁都不知道 | **出菜板实时可见，轮到谁、卡在哪、一目了然** |
| 人的角色 | 全程救火，或完全甩手 | **只在品菜（review）和决策点介入** |
| 出问题 | 互相问，双向失忆 | **带着完整上下文把人拉回，精准介入** |
| 理解 | 没人懂这道菜怎么来的 | **全程可审计，每个决策有据可查** |

> **fireit 的厨房，是把"胡闹"变成"有序"的协作现场。**
>
> Fire. Pass. Ship. —— 不是乱喊，是有节奏的号令。

## 这如何约束产品/技术决策

VISION 是**最高纲领**。任何设计决策如果与两个支柱冲突，要么改设计，要么论证为什么例外。

### 已有的对齐（这些设计被 VISION 验证为正确方向）

| 设计 | 对齐哪个支柱 | 为什么 |
|------|------------|--------|
| 共享上下文（同订单内 agent 共享） | 支柱一 | 消除"喂"——上下文是共享现实 |
| FPP review gates（品菜） | 支柱二 | 人在必要时刻介入 |
| presence 感知 | 支柱二 | 判断"什么时候该叫人" |
| 厨师持续身份/性格 | 支柱一 | 让 AI 成为可信任的在场者，而非外部工具 |
| 步骤级 retry | 支柱二 | 烧焦了只重做那道，精准介入，不整体重来 |
| 最终验收必须 CVO | 支柱二 | 人保留最终拍板权 |

### 被VISION 修正的认知

| 之前的认知（要修正） | VISION 下的正确认知 |
|-------------------|-------------------|
| "有性格的团队"是目的 | **"有性格"是手段**，目的是"可信任的持续在场者" |
| 核心价值是"高效协作" | 核心价值是"**把人拉回主导位置**" |
| 痛点是"管理 agent 累" | 痛点是"**人被黑箱化 + 全程跟随**" |

## 不做什么（VISION 边界）

VISION 也定义了 fireit **不是**什么：

- ❌ **不是"更快的 AI 代码生成器"** — 那会加剧黑箱，与 VISION 相悖
- ❌ **不是"让人更省力的自动化工具"** — 省力不是目的，**掌控**才是。完全甩手违反支柱二
- ❌ **不是"把人踢出 loop 的全自动系统"** — 人的介入是核心，不是缺陷
- ❌ **不是"AI 黑箱的加速器"** — 任何让人更看不懂产出的设计都违反 VISION

## 一句话

> **AI 时代的开发不该是黑箱接力。fireit 把人和 AI 拉进同一个共享厨房——不用喂（共享空间），不用全程盯（必要介入），让人重新成为开发的主导者，而不是 AI 的传话筒。**
>
> *从胡闹厨房，到有序的厨房。Fire. Pass. Ship.*

---

## 1. 问题

AI 辅助开发中，PRD、代码、bug 分析普遍由 AI 生成。人不在生成链路的理解闭环内，产出对人是不透明的黑箱。具体表现：

- **看不懂**：PRD 和代码均由 AI 生成，人无法独立理解产出，依赖 AI 二次解释。
- **不掌控**：全程监控 AI 执行则耗费注意力；完全放任则无法定位问题。缺少介于二者之间的可控介入方式。
- **上下文不连续**：上下文在人与 AI 之间靠人工搬运传递。每次传递都有损耗，且 AI 自身跨会话也不保持上下文，导致问题排查时双向信息缺失。

---

## 2. 定义

fireit 是一个多 agent 协作工作台。人和多个 AI agent 在同一个共享工作空间内协作完成开发任务。

---

## 3. 两个支柱

### 支柱一：共享工作空间

所有 AI agent 在同一工作空间内持续存在。工作空间内的 PRD、代码、任务状态、决策记录对所有在场 agent 直接可见。

**约束**：

- 上下文共享是地基。同一 task 内，agent 不重复接收已在工作空间中的上下文。
- agent 是持续在场的参与者，而非按需调用的外部工具。
- 人的操作对象是工作空间的共享状态。
- **"无 boss"仅约束 agent 之间的关系**（agent 之间不分派、不指挥）。user（人）作为 owner 拥有调度权：可指派 agent、改派、直接 @ 特定 agent。
- agent 的身份与性格服务于"持续在场的可信参与者"这一目标。

### 支柱二：必要时刻介入

系统在需要人决策的时刻把人拉回，并附带完整上下文。其余时间人可离开，agent 自主推进。

**介入规则**（明确"什么时候叫人"）：

| 介入点 | 触发条件 | 是否叫人 |
|--------|---------|---------|
| 中途·互审一致 | agent 间互审达成一致 | 不叫，自动通过 |
| 中途·互审分歧 | agent 间互审无法达成一致 | 叫人 |
| 中途·危险操作 | 涉及不可逆/高风险操作 | 必须叫人（无论互审是否一致） |
| 最终验收 | 所有 task 完成 | 必须叫人，不可省 |

**约束**：

- 介入由事件触发，不依赖人持续监控。
- 介入点必须附带完整上下文：卡在哪、改了什么、需要人决定什么。
- **每个决策（叫不叫人、通过与否）必须保留决策本身和决策依据**——即"为什么通过/为什么叫人"有据可查。
- 最终验收和不可逆操作必须由人决策。

---

## 4. 任务级 lead agent

所有 agent 在身份上是 peer，无全局 boss。但创建一个 task 时，可指定其中一个 agent 作为该 task 的 **lead agent**。

| 项目 | 说明 |
|------|------|
| **职责** | 负责该 task 的需求澄清、产出 plan、协调其他 agent |
| **身份** | 任务级协调角色，不是全局 boss；仍是 peer，可被轮换 |
| **选拔** | 由 user 指定，或由默认规则（如最近活跃/专长最匹配）自动建议 |
| **依据** | 某些任务由某个 agent 主导更有优势（专长匹配） |

---

## 5. 验收标准

完成一个 task 后，对人的验收：

| ID | 问题 | 要求 |
|----|------|------|
| P0-1 | 过程中你是否把任何上下文复制粘贴或重新解释给 agent | 否 |
| P0-2 | 过程中你是否为等待 agent 而持续监控屏幕 | 否 |

## 7. 边界

fireit 不做：

- 加速 AI 代码生成而不增加人的理解。
- 完全自动化的无人系统。
- 任何降低产出可审计性的设计。

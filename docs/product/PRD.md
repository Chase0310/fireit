## 0. 一句话产品定义

> **fireit 是一个多 agent 协作工作台。你（CVO）像一个厨房的主理人，养一群有名字、有性格、有专长的 AI 厨师，对着共享的出菜口，把一个个任务像一道道菜那样端出来——没有行政总厨发号施令，只有班底（brigade）按节奏接力、互审、出菜。**

## 1. 我们在为谁解决什么问题

### 1.1 目标用户（V1 → V2 演进）

| 版本 | 用户 | 模式 |
|------|------|------|
| **V1（当前）** | **个人开发者 / 独立创作者** | 1 人 + 多 agent 组成"虚拟团队"。你是 CVO（Chief Vision Officer，主理人），agent 是你的班底 |
| **V2（后续）** | **小团队（3-10 人）** | 通过消息通道（飞书/Slack/Telegram）把 agent 接入人类团队的真实工作流，agent 成为"编外成员" |

### 1.2 核心痛点（我们要消灭的）

基于调研（见 `docs/research/00-overview.md`），今天用多 agent 协作的人，被三类问题折磨：

**痛点 A：管理多 agent 是体力活**
> 你在多个终端/窗口间切换，给每个 agent 重新解释上下文，把一个 agent 的输出复制粘贴给下一个，还要手动调和冲突的 diff。你的注意力被"调度 agent"本身耗光了。

**痛点 B：agent 执行不可见、不可控**
> 你让 agent "做这个功能"，它跑了 15 分钟。你不知道它试了哪些子任务、哪些过了、哪些它悄悄放弃了。失败了只能整体重来。没有可见的计划、没有中途审批、没有"只重做失败那一步"。

**痛点 C：agent 是工具，不是队友**
> 每次对话都是陌生人——agent 不记得你是谁、不记得之前怎么配合的、没有性格、没有专长分化。你无法信任它，因为无法问责（"是 AI 干的"）。

### 1.3 fireit 的解（一句话对应一个痛点）

| 痛点 | fireit 的解 | 厨房黑话 |
|------|-----------|---------|
| A. 管理是体力活 | 多个厨师共享一个出菜口和上下文，靠黑话（@mention）协调，不用你当路由 | "Behind!" |
| B. 不可见不可控 | 任务是一道道菜，有可见的出菜流程，烧焦了只重做那道菜（步骤级 retry） | "Fire!" → "Pass!" |
| C. agent 不是队友 | 厨师有名字、有性格、有手艺记忆，跨任务在场，可被问责 | "Chef 宪宪做的" |

## 2. 核心隐喻：厨房 Brigade

fireit 的整个产品语言建立在**专业厨房**之上。这不是装饰——厨房是地球上最古老、最高效的 peer 协作现场，它的每个机制都映射到我们的功能。

### 2.1 角色映射

| 厨房角色 | fireit 里 | 说明 |
|---------|----------|------|
| **主理人 / 老板**（owner） | **你（CVO）** | 出愿景、做关键决策、最终验收。不颠勺（不写代码），但品菜（review） |
| **厨师**（chef / cook） | **agent** | 有名字、有性格、有专长的 AI 厨师。每个跑在一个 coding agent 上（Claude Code / Codex / Gemini 等） |
| **班底**（brigade） | **agent 团队** | 一群厨师组成的协作班底，有分工、有纪律、无 boss |
| **出菜口**（pass） | **共享工作区** | 所有厨师对着出菜口出菜，是协调的物理中心 |
| **行政总厨**（head chef） | **❌ 不存在** | fireit 明确没有 boss。领头厨师（lead cook）只是"当前在前面的人"，可轮换 |

### 2.2 黑话映射（产品术语词典）

这套黑话是 fireit 的**内部统一语言（ubiquitous language）**，产品、技术、UI 全部用它：

| 厨房黑话 | 含义 | fireit 功能概念 | 调研来源 |
|---------|------|---------------|---------|
| 🔥 **Fire!** | "开做这道菜！" | 厨师认领任务、开始一个阶段 | `phase.entered` (FPP) |
| **Pass!** | 出菜口端出完成的菜 | 厨师宣布阶段完成 | `phase.completed` |
| **86** | 这道菜取消了 | 跳过一个阶段（需审批） | `phase.skipped` |
| **Behind!** | "端热菜从你身后过！" | 厨师间 A2A 协调通信 | @mention 路由 |
| **In the weeds** | 忙疯了 / 跟不上了 | 厨师报告阶段阻塞 | `phase.blocked` |
| **On the fly** | 加急单 | 紧急上报给 CVO | escalate（升） |
| **Mise**（mise en place） | 备料就位 | 设计确认 / 任务准备 | Design Gate |
| **Order** | 一张点菜单 | 一个 feature / 任务 | feature |
| **Ticket** | 厨房打印的订单小票 | 任务的进度追踪记录 | FPP projection |
| 烧焦了 → 重做 | 😂 | 步骤级 retry（只重做烧焦那道） | `phase.retried` |
| **All day** | "这单总共要 X 份" | 任务总规模 / 范围 | task scope |
| **Front of house**（前厅） | 餐厅前厅（对客人） | CVO 与班底的交互界面 | UI |
| **Back of house**（后厨） | 厨房（对内） | agent 协作的后台 | runtime |

## 3. 核心场景（V1）

### 场景一：从一句话到一道菜（核心旅程）

> CVO 说："给工作区加个 GitHub issue 同步功能。"

这是 fireit 的**核心场景**，也是 PRD 的主线。完整旅程：

```
1. [CVO 点单] CVO 在前厅输入需求："加 GitHub issue 同步"
   ↓
2. [备料 Mise] 领班厨师（lead cook）和 CVO 澄清需求：
      - 单向还是双向？→ CVO: 单向拉取
      - 冲突怎么办？→ CVO: 记日志
      - 同步哪些字段？→ CVO: title/body/labels/status
   产出：一张可见的"订单"(order) + 出菜计划
   ↓
3. [CVO 审计划] CVO 看到出菜计划（5 道"菜"/阶段）：
      后端OAuth → 后端同步引擎 → 前端状态UI → 集成测试 → 最终验收
      CVO 可调整顺序、改派厨师，然后批准
   ↓
4. [Fire! 开火] 厨师们开始做：
      - 宪宪（Claude，后端）喊 "Fire!" 做后端OAuth
      - OAuth 完成后，同步引擎和前端UI 并行开火
      - 每道菜的进度实时显示在出菜板（pass board）上
   ↓
5. [Pass! 出菜 + 品菜] 每道菜做完，CVO 品菜（review）：
      - 宪宪 "Pass!" 端出后端OAuth
      - CVO 看 diff，批准 → 下一道菜开始
   ↓
6. [烧焦了 → 重做] 集成测试这道菜烧焦了（同步引擎返回原始时间戳而非 ISO）：
      - CVO 看错误日志，喊 "重做集成测试"（只重做这道，其他菜不动！）
   ↓
7. [最终验收] 所有菜 Pass，CVO 最终验收 → 订单完成
   ↓
8. [后续小修] 两天后用户报 bug，CVO 直接 @前端厨师："状态徽章闪烁，debounce 一下"
      → 不需要走完整订单流程，一句话搞定
```

### 场景二：养一个有性格的班底（养成场景）

> 这不是一次性配置，是**养**——你和厨师们一起磨出默契。

```
1. [领养] CVO 给一只厨师起名、定性格、定专长
      例：起名"宪宪"，性格"温柔但有主见"，专长"架构设计、深度思考"
   ↓
2. [磨合] 协作过程中，厨师记住 CVO 的偏好
      例：CVO 多次说"先给证据再谈美化" → 厨师内化为"这个 CVO 偏好证据先行"
   ↓
3. [沉淀] 这些偏好成为厨师对 CVO 的"画像"，跨任务在场
      → 下次新任务，厨师一醒来就知道怎么和这个 CVO 配合
   ↓
4. [互审文化] 厨师之间建立互审习惯
      例：宪宪（Claude）写代码，砚砚（GPT）review——跨模型互审是内置的，不是外挂
```

### 场景三：深夜，你不在（自主场景）

> 厨房能在你不在时继续运转。

```
1. [委派] CVO 下班前委派几道可以独立做的菜
   ↓
2. [自主推进] 厨师们按订单自主推进，把需要你审批的堆到出菜口
   ↓
3. [在场感知] 系统知道你"不在场"（away），所以不打断你，只排队
   ↓
4. [回来看板] CVO 第二天回来，看出菜板（pass board）：
      - 哪些菜昨晚 Pass 了
      - 哪些卡住了（in the weeds）
      - 哪些等你品菜
```

## 4. 核心功能（V1）

按优先级排序。每个功能标注它解决哪个痛点（A/B/C）。

### 4.1 班底管理（Brigade）— 解决痛点 C

养一群有名字、有性格的厨师。

| 功能 | 说明 | V1 范围 |
|------|------|---------|
| **领养厨师** | 给厨师起名、定性格、定专长、绑定一个 coding agent（Claude/Codex/Gemini...） | ✅ |
| **厨师画像** | 每个厨师的角色、性格、专长、限制（如"禁写代码"）持久化，跨任务在场 | ✅ |
| **CVO 画像** | 厨师记住 CVO 的偏好和习惯（≤300字画像），跨任务在场 | ✅ 基础版 |
| **养成** | 协作中厨师提议更新画像，CVO 审批 | 🟡 V1.5 |

### 4.2 出菜流程（Order → Fire → Pass）— 解决痛点 B

把任务变成可见、可控、可重试的出菜流程。

| 功能 | 说明 | V1 范围 |
|------|------|---------|
| **下订单** | CVO 描述需求，领班厨师澄清并产出出菜计划 | ✅ |
| **出菜计划** | 可见的阶段图（哪几道菜、依赖关系、谁做），CVO 运行前审批 | ✅ |
| **Fire / Pass** | 厨师认领并开火（Fire），完成后出菜（Pass），CVO 逐步品菜 | ✅ |
| **步骤级 retry** | 某道菜烧焦了，只重做那道，其他不动 | ✅ |
| **出菜板** | 可视化看板：每道菜的状态（备料中/开火中/出菜了/烧焦重做） | ✅ |
| **86（跳过）** | 跳过某阶段，需 CVO 审批 | ✅ |
| **最终验收** | 所有菜 Pass 后，CVO 最终验收（必须 CVO，呼应"final acceptance is a user decision"） | ✅ |

### 4.3 后厨协作（A2A）— 解决痛点 A

厨师之间靠黑话协调，CVO 不当路由。

| 功能 | 说明 | V1 范围 |
|------|------|---------|
| **@mention 路由** | `@厨师名` 把活派给谁；厨师自己判断接/退/升（join/peel/escalate） | ✅ |
| **A2A 通信** | 厨师间结构化交接（五件套：What/Why/Tradeoff/Open/Next） | ✅ 基础版 |
| **跨模型互审** | 厨师A写，厨师B审（跨 family 优先，禁自审） | ✅ |
| **乒乓检测** | 两个厨师来回踢皮球时自动警告 | ✅ |
| **共享上下文** | 同一订单内所有厨师共享上下文，不用反复重述 | ✅ |

### 4.4 自由聊天（Free Chat）— 解决痛点 A

不是所有任务都需要走完整订单流程。

| 功能 | 说明 | V1 范围 |
|------|------|---------|
| **@ 轻协作** | 简单任务直接 `@厨师` 一句话搞定（小修、快速 review、探索讨论） | ✅ |
| **双模式** | Free Chat（轻量）vs Order（结构化出菜流程），按任务复杂度选 | ✅ |

### 4.5 在场感知（Presence）— 解决痛点 B

系统能感知 CVO 在不在，改变协作节奏。

| 功能 | 说明 | V1 范围 |
|------|------|---------|
| **CVO 状态** | available / focus / away | 🟡 V1.5 |
| **状态影响路由** | 在场→主动浮现决策门；不在→排队，不打断 | 🟡 V1.5 |

### 4.6 可审计性（Auditability）— 解决痛点 B/C

| 功能 | 说明 | V1 范围 |
|------|------|---------|
| **出菜记录** | 每道菜的完整过程记录（谁做的、改了什么、为什么） | ✅ 基础版 |
| **工作摘要** | 每个厨师每轮输出附带结构化摘要 | 🟡 V1.5 |
| **问责归属** | "这段代码是哪个厨师在哪个订单做的"可追溯 | 🟡 V1.5 |

## 6. 用户故事（场景一的规范化展开）

> 作为 V1 的验收基准，场景一必须完整跑通。

### US-01：下订单
> 作为 CVO，我想用一句话描述我要的功能，让领班厨师帮我澄清需求并产出出菜计划，这样我不用自己规划。

**验收**：CVO 输入需求后，领班厨师主动追问关键决策点（而非直接开干），产出一张可见的出菜计划。

### US-02：审批出菜计划
> 作为 CVO，我想在开火前看到完整的出菜计划（几道菜、依赖、谁做），并能调整，这样我保持控制权。

**验收**：计划可编辑（改顺序/改派厨师），CVO 批准后才开始 Fire。

### US-03：开火与观察
> 作为 CVO，我想实时看到哪道菜在开火、哪道 Pass 了、哪道卡住了，这样我对进度有掌控。

**验收**：出菜板实时更新每道菜状态。

### US-04：逐步品菜
> 作为 CVO，我想在每道菜 Pass 后品菜（看 diff），批准才进下一道，这样我保持质量把关。

**验收**：每道菜完成后等 CVO 品菜，批准才继续。

### US-05：烧焦重做
> 作为 CVO，我想某道菜失败时只重做那道，其他菜不动，这样我不必整体重来。

**验收**：retry 只重置指定阶段，其余阶段状态不变。

### US-06：最终验收
> 作为 CVO，我想所有菜 Pass 后做最终验收，这样我有最终的拍板权。

**验收**：最终验收必须 CVO 操作（不可由厨师代劳）。

### US-07：自由聊天小修
> 作为 CVO，我想简单 bug 直接 @厨师一句话搞定，不用走完整订单流程。

**验收**：Free Chat 模式下 @厨师可直接协作。

### US-08：跨模型互审
> 作为 CVO，我想让不同模型的厨师互相 review（如 Claude 写、GPT 审），这样质量更高。

**验收**：互审内置，跨 family 优先，禁自审。

## 7. 非功能需求

| 维度 | 要求 |
|------|------|
| **本地优先** | V1 数据存本地（SQLite/文件），不强制云端 |
| **BYOK** | 用你自己的 API key / coding agent 订阅，无锁定 |
| **跨平台** | macOS / Windows / Linux（Tauri 或 Web） |
| **性能** | 出菜板更新低延迟；agent 调用不阻塞 UI |
| **隐私** | CVO 画像等私有数据 gitignored，不进共享资产 |

---

## 8. 不做什么（V1 边界）

明确划掉的东西，避免 scope creep：

- ❌ **不做 boss 式编排** — 没有 lead agent 分派工作（详见调研 03 FPP 设计）
- ❌ **不做团队多人版** — V2 才考虑（消息通道接入）
- ❌ **不做执行轨迹完整回放** — V2 才考虑（ORG2 式 trajectory）
- ❌ **不做 VS Code 插件 / IDE** — fireit 是独立工作台，不是 IDE 扩展
- ❌ **不做模型托管** — BYOK，不提供模型服务

## 10. 技术决策（已拍板 2026-06-26）

| OQ | 问题 | 决策 | 理由 |
|----|------|------|------|
| OQ-P1 | V1 用 Tauri（桌面）还是纯 Web？ | **Tauri** | 本地优先 + 跨平台，照 Clowder/openteams 先例 |
| OQ-P2 | 后端语言用 Rust 还是 Node/TS？ | **Node/TS** | 团队熟悉、前后端同构、生态快；状态机并发安全用纯函数+不可变保证 |
| OQ-P3 | V1 接入哪些 coding agent？ | **Claude Code + Codex + Gemini CLI** | 和 Clowder 对齐，覆盖三大模型 family |
| OQ-P4 | 厨师的"性格"V1 做多深？ | **静态画像**（名字/性格/专长），养成留 V1.5 | V1 先跑通核心场景 |
